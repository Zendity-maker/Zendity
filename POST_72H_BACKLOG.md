# Backlog Post-72h - Zendity Clinical Care Piloto

## Área de Producto: Experiencia Clínica (Tablet-First)

### 🎫 TICKET: Refinamiento de UX/UI First-Party (Nivel Crítico)
**Origen del ticket**: Hallazgo de primeras 24 hrs en piso.
**Categoría**: Refactorización de Interfaz / UX Operativa

#### El Problema
La interfaz en tabletas para el personal clínico no está optimizada primariamente para tacto rápido (tablet-first), sino que funciona como una versión *responsive* de desktop. Esto impacta negativamente:
1. **Adopción de Usuarios Base**: Fricción de la curva de aprendizaje ante interfaces densas.
2. **Velocidad de Gestión**: Zonas táctiles pequeñas propensas a *miss-clicks* o necesidad de excesivo scroll y precisión.
3. **Fatiga del Personal**: Carga visual y operativa innecesaria para el entorno de un turno largo de cuidados clínicos.

#### Requerimiento para Sprint Post-72h
Llevar la aplicación `Zendity Clinical Care` hacia una línea de diseño genuinamente **Tablet-First**:
- **Zonas Táctiles (Touch Targets)**: Aumentar el tamaño mínimo de botones interactivos según estándares de accesibilidad (WCAG ~44x44px o superior en contextos críticos).
- **Densidad de Información**: Reducir el ruido visual en las pantallas de acción clave (Ej: Handover, eMAR, bitácoras de signos vitales).
- **Navegación Fluida (Gestures)**: Priorizar interacciones que reduzcan la cantidad de *taps* necesarios para ejecutar acciones repetitivas.
- **Claridad de Estados**: Feedback visual inmediato post-acción (toque, swipe, envío exitoso).

#### Prioridad
🔥 **Alta/Crítica**. Esta línea de trabajo será el enfoque primario de diseño una vez superada la ventana de monitoreo de estabilidad funcional inicial (72h).
